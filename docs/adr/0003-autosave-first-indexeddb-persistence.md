# Autosave-first IndexedDB persistence

The MVP will use autosave-first PWA persistence backed by IndexedDB through Dexie for app-owned Notebook data, with notebook export/import for backup. Persisted Notebook data and Notebook Export should use the same versioned Zod schema and migration pipeline rather than separate contracts or a raw database dump. Persisted Notebook data must include a `schemaVersion`, Zod validation at the persistence boundary, and explicit migrations from the first stored shape, because TypeScript does not protect old browser data or imported backups. Autosave failures must be visible, sticky, and retryable; the UI must not report a false saved state after a failed write. If stored data cannot be validated or migrated, the app should show a recovery screen, offer raw payload export for backup/debugging, and let the user choose a new Notebook or import rather than silently resetting. If the same Notebook changes in another browser tab, the MVP should detect the newer external write and avoid silent last-writer-wins rather than attempting real-time merge. The MVP will not encrypt IndexedDB data at rest, so the UI should use clear local-browser-storage risk language; passphrase-protected Notebook Export can be considered later. Persisted Pages should keep stable domain-owned Canvas Items and Canvas Regions alongside a canvas-library snapshot boundary, with any canvas-library persistence kept behind a Page canvas adapter rather than treated as the main storage boundary. The Local Index should use rebuildable tables in the same Dexie database, while Notebook Export should carry source Notebook data rather than derived index state. Manifest metadata can arrive early, but service worker caching should wait until the React, TypeScript, and Dexie path is stable. This favors seamless rough-work capture over file-first storage, while avoiding raw IndexedDB code and leaving File System Access integration as a later enhancement.
