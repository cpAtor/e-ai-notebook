# Extensible canvas foundation

The MVP will build on an extensible canvas library in the tldraw direction rather than embedding Excalidraw, building a custom canvas from scratch, or shipping a frameworkless DOM shell that will be replaced later. The application shell should be React and TypeScript from the start, with Notebook behavior kept in a testable domain/application layer outside React and exposed through a small UI boundary. Source modules should separate domain, persistence, canvas adapter, UI, and app composition responsibilities so React and tldraw remain adapters around the Notebook domain. A slice that claims to deliver the Page canvas should include a real tldraw-backed canvas rather than a textarea or other placeholder, starting with a narrow text Canvas Item vertical slice instead of every Canvas Item type. Mobile and touch canvas interaction are first-class MVP requirements, not a later-only desktop afterthought. The first Text Canvas Item should use built-in tldraw text shapes with an app-owned mapping layer when possible, reserving custom shapes for item types that need them. Domain entities should use app-owned opaque prefixed IDs; Canvas Region references should use app-owned Canvas Item IDs, Page IDs, and spatial bounds rather than raw tldraw shape IDs alone. The canvas adapter should sync tldraw edits into domain Canvas Items on debounced persistable changes, and app load should validate/diff canvas, domain, and indexable state in the background so interrupted sessions can recover consistency. Notebook, Page, Canvas Item, Canvas Region, and persistence boundaries should be shaped around the actual canvas integration instead of being implemented twice. The product needs a rough-work feel, but stable custom items, region navigation, and indexing hooks are more important than maximizing sketch aesthetics in the first version.
